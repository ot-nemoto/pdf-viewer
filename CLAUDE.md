# CLAUDE.md

## 作業開始時のチェックリスト

1. `docs/product.md` を読みプロダクトの目的・スコープを理解する
2. `docs/ui.md` で今回のタスクに関連する画面・機能挙動の契約を確認する
3. `docs/architecture.md` で実装方針・技術スタック・非機能要件を確認する
4. タスクの一覧・状態・実装順は GitHub Issues / Milestone で確認する

## ドキュメント構成ルール

### 中核原則（source of truth）

- **コードが正の内容はドキュメント化しない**（ディレクトリ構成・コンポーネント一覧・スクリプト表・依存バージョン数値・設定ファイル本文・CI ステップ本文）。乖離時はコードを正とする。
- **ドキュメントに残すのは「意図・契約・規約」**（目的・意思決定理由・UI 規約・画面/機能挙動の契約・運用手順・バージョン gotcha）。
- バージョンは常に `package.json` を正とし、ドキュメントに数値を書かない。

### 必須ドキュメント

実装・コード生成を行う前に、以下のドキュメントが `docs/` 配下に揃っていることを確認すること。
不足しているドキュメントがある場合は、作業開始前にユーザーに確認または作成を促す。

| ファイル | 役割 | 主なセクション |
|----------|------|--------------|
| `docs/product.md` | プロダクトの目的・対象ユーザー・スコープ | 目的、対象ユーザー、成功指標、スコープ外 |
| `docs/architecture.md` | 技術スタック・実装方針・非機能要件 | 技術スタック、状態管理、非機能要件、バージョン gotcha |
| `docs/ui.md` | 画面遷移・表示状態・機能挙動の契約・UI 規約 | 画面遷移図（Mermaid）、表示状態（Loading / Empty / Error）、機能挙動仕様、UI 規約 |
| `docs/development.md` | 開発・運用手順 | ローカルセットアップ、デプロイ/リリース手順、依存更新 |

> 機能要件・非機能要件は `ui.md`（機能挙動の契約）／`architecture.md`（非機能）に統合する。タスク管理は GitHub Issues / Milestone、テスト方針は本ファイルの「テストルール」、手動確認は「UI実装の動作確認ルール」で扱う（個別ドキュメントは持たない）。

### 条件付き必須ドキュメント

本プロジェクトは **サーバーを持たないクライアント完結の SPA** であり、外部 REST API の提供・DB 永続化・認証フロー・外部サービス連携・独自インフラ構成のいずれも該当しない。
そのため `api.md` / `actions.md` / `schema.md` / `auth.md` / `integrations.md` / `infra.md` は作成対象外とする。将来これらの実装が発生した場合は、その時点で作成する。

### README.md

`README.md` はリポジトリルートに配置し、GitHub のトップページに表示されるランディングページとして機能させる。

**記載する内容（他のドキュメントと重複させない）：**

| セクション | 内容 |
|-----------|------|
| プロダクト名・概要 | 一文でプロダクトの目的を説明 |
| 主な機能 | 主要機能の箇条書き（詳細は `docs/product.md` / `docs/ui.md` に委譲） |
| ドキュメント | `docs/` 配下の全ドキュメントへのリンクテーブル |
| クイックスタート | 起動に最低限必要なコマンドのみ。詳細は `docs/development.md` へのリンクで委譲 |

## ドキュメント更新ルール

- 実装中に仕様変更が発生した場合、コード修正と同時に該当ドキュメントを更新する
- ドキュメントとコードの乖離が生じた場合は、原則ドキュメントを正として扱いコードを修正する。ただし意図的な仕様変更の場合はドキュメントを先に更新してからコードを変更する

## テストルール

### テスト種別

| 種別 | ツール | 対象 |
|------|--------|------|
| ユニットテスト | Vitest + Testing Library（jsdom） | ロジックを集約する hooks・純粋なユーティリティ関数 |
| 手動動作確認 | 実機（ブラウザ） | UI 表示・操作・エラー表示（「UI実装の動作確認ルール」参照） |

### 完了条件

- **ロジックを集約する hooks（`src/hooks/`）はユニットテストの作成をもって完了とする**
- **純粋なユーティリティ関数もユニットテストの作成をもって完了とする**
- UI コンポーネントのユニットテストは必須としない（表示・操作は手動動作確認でカバー）

### カバレッジ方針

- ロジックの分岐（境界値・クランプ・状態遷移）を優先的にカバーする（例: `usePdfFile` のページ送りクランプ、ズーム上下限、自動送りトグル、最終ページでの再生開始、closeFile によるリセット）
- カバレッジ率の数値目標は設けず、「ロジックの分岐が検証されていること」を基準とする

### 実行

- ローカル: `npm run test`（`vitest run`）。設定・setup は `vite.config.ts` の `test` と `src/test/setup.ts` を正とする
- CI: `.github/workflows/ci.yml`（PR で lint + test）を正とする

## UI実装の動作確認ルール

- **UI を含む実装はアプリを起動した手動動作確認をもって完了とする**
- 実装完了後、コミット・プッシュの前に動作確認一覧を **Issue にコメントとして投稿** し、ユーザーに提示する
- ユーザーが動作確認を実施し、問題がなければコミット・プッシュに進む
- 動作確認中に一覧の追加・変更が発生した場合は、**新しいコメントを追加せず、最初に投稿したコメントを編集して更新する**

### 動作確認チェックリストの形式

以下の観点を網羅した表形式で投稿する：

| # | 手順 | 確認観点 | 期待値 |
|---|---|---|---|
| ... | ... | ... | ... |

### 最低限カバーすべき観点

- **画面表示**: データが正しく表示されるか、空状態（未読込）の表示
- **インタラクション**: ボタン・キー操作・フォームの操作、状態変化
- **エラー**: 不正な入力（PDF 以外）・読み込み失敗時の表示

## バージョン準拠ルール

- フレームワーク・ライブラリのバージョンは **常に `package.json` を確認して**、そのバージョンの仕様に従う
- 古いバージョンの知識や一般的な慣習を根拠にしない
- 外部ツール（Copilot 等）やレビュアーからフレームワーク・ライブラリの慣習に関する指摘を受けた場合、`package.json` のバージョンに基づく仕様を確認してから採否を判断する
- バージョン固有の仕様・既知の指摘パターンは `docs/architecture.md` に記載する

## コーディングルール

- lint / format は **Biome** に統一する（`npm run lint` / `npm run format`）。ESLint / Prettier は使わない
- コメント: 自明でないロジックにのみ付ける（過剰なコメント不要）
- エラーハンドリング: 外部入力（ドロップされたファイル）・ライブラリの読み込み結果など境界でのみ行う
- 言語・ツール・フォーマッタなどプロジェクト固有の選択は `docs/architecture.md` を参照

## GitHub Issues 管理ルール

新規タスク・不具合はすべて GitHub Issues で管理する。

### ブランチ・マージルール

- 作業は `develop` ブランチで行い、`develop` および `master` へのマージは **必ず PR を作成**し、ユーザーの承認を得てからマージする
- 直接マージ（`git merge`）や master への直接コミットは行わない

### Issue 作成ルール

| 項目 | ルール |
|------|--------|
| タイトル | `[T番号] タスク名` の形式（例: `[T6] XXX機能追加`） |
| T番号 | 直前の最大番号 + 1 で採番する |
| Label | `enhancement`（新機能）/ `fix`（不具合）/ `testing`（テスト）/ `documentation`（ドキュメント）/ `ci`（CI/CD・GitHub Actions・Dependabot などインフラ寄りの変更）から選ぶ |
| 本文 | 概要・要件・依存 Issue を記載する |

### Issue クローズルール

Issue をクローズする際は、**必ず関連するコミット ID（またはマージコミット）をコメントに含める**。

```bash
gh issue close <番号> --comment "完了。実装コミット: <short-hash> (<コミットメッセージ>)"
```

## バージョニングルール

`package.json` の `version` はセマンティックバージョニング（SemVer）に従う。

> **バージョンバンプは `bump-version.yml` ワークフローが自動で行う。**
> master への PR にラベルを付けるだけでよく、`package.json` の手動更新は不要。

| ラベル | バージョン操作 | 例 |
|--------|-------------|-----|
| `bump:minor` | マイナー +1、パッチ → 0 | `0.1.0` → `0.2.0` |
| `bump:patch` | パッチ +1 | `0.2.0` → `0.2.1` |

| 変更の種類 | 使用ラベル |
|-----------|-----------|
| 新機能追加 | `bump:minor` |
| バグ修正・小改善のみ | `bump:patch` |

`develop` → `master` の PR は `auto-pr-to-master.yml` が日次で自動作成し、含まれる Issue のラベルから `bump:minor/patch` を自動判定する。master への push で `release.yml` が tag と GitHub Release を自動生成する。

## Git 操作ルール

- `git commit` / `git push` はユーザーの **明示的な承認（「OK」など）を受けてから** 実行する
- 実装・ドキュメント更新が完了したら、変更内容をユーザーに提示して承認を待つ
- 承認前にコミット・プッシュを自動で行わない

### Issue 着手フロー

1. Issue の仕様・現状コードを確認し、作業タスクを整理する
2. **Issue にコメントとして**作業タスク（チェックリスト形式）を投稿する
3. ユーザーに確認・承認を得る
4. `feature/t<番号>-<概要>` の形式でブランチを切る
5. 承認後に実装を開始する

> Issue 本文は「何をするか（仕様）」、コメントは「どうやるか（作業計画・進捗）」として使い分ける。

### Issue 単位の作業フロー

1. 変更内容をユーザーに提示して承認を待つ
2. 承認後、`git commit` → `git push` を実行する
3. PR を作成し、ユーザーのレビューを待つ（マージは自分でしない）
4. マージ後、対応する GitHub Issue を関連コミット ID を添えてクローズする

### 作業スコープルール

- **指示されたタスク（Issue）のみを実行する**。ユーザーが明示的に指定していないタスクは、自己判断で着手しない
- 複数 Issue をまとめて依頼された場合も、1 Issue ごとに完了・承認を経てから次に進む

## PR レビュー指摘の対応ルール

- Copilot・レビュアーからの指摘は、内容を精査し **妥当性の有無・概要・対応案（対応不要ならその理由）を各指摘コメントに返信**する
- 指摘が複数ある場合は、まとめて対応せず **各指摘ごとに個別に対応**する（個別コミット・個別返信・個別 Resolve）
- 対応不要と判断した場合はその理由をコメントで返す
- 指摘対応のコミットメッセージは `fix: <内容>` とする
- 指摘に対応したら、該当コメントに対応済みの旨をコミット ID を添えて返信し、スレッドを Resolve する

```
対応済み。修正コミット: <short-hash> (<コミットメッセージ>)
```
